# Extensions

## Goal

Understand PostgreSQL's extension system and learn the extensions you'll use with Supabase — especially `pgvector` for semantic search.

---

## 1. What Are Extensions?

Extensions are **add-on modules** that add new types, functions, operators, or indexes to PostgreSQL. They're like plugins.

```sql
-- List all available extensions
SELECT name, comment FROM pg_available_extensions ORDER BY name;

-- List installed extensions
SELECT * FROM pg_extension;
-- Or in psql:
\dx
```

### Install / Uninstall

```sql
-- Install an extension
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Remove an extension
DROP EXTENSION IF EXISTS "uuid-ossp";
```

> **In Supabase**, many extensions are pre-installed or available. You enable them with `CREATE EXTENSION` in a migration.

---

## 2. Essential Extensions

### uuid-ossp — UUID generation

Provides additional UUID generation functions beyond `gen_random_uuid()`.

```sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- v4: random UUID (same as gen_random_uuid())
SELECT uuid_generate_v4();

-- v1: timestamp-based UUID (includes time + MAC address)
SELECT uuid_generate_v1();
```

> **In PostgreSQL 13+**, `gen_random_uuid()` is built-in — you don't need `uuid-ossp` for random UUIDs. But `uuid-ossp` provides other variants (v1, v3, v5).

---

### pgcrypto — Cryptographic functions

Hashing, encryption, and random data generation.

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Hash a password with bcrypt
SELECT crypt('my_password', gen_salt('bf'));
-- Result: $2a$06$... (bcrypt hash)

-- Verify a password
SELECT crypt('my_password', stored_hash) = stored_hash AS is_valid;

-- Generate random bytes
SELECT gen_random_bytes(16);

-- Generate random UUID (alternative to gen_random_uuid)
SELECT gen_random_uuid();
```

> **Supabase Auth handles password hashing for you.** But `pgcrypto` is useful for hashing API keys or generating tokens in custom functions.

---

### pg_trgm — Trigram similarity search

Enables fuzzy text search — finding "similar" strings even with typos.

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- Similarity score (0 to 1)
SELECT similarity('PostgreSQL', 'postgres');  -- ~0.5
SELECT similarity('hello', 'helo');           -- ~0.6

-- Find similar text
SELECT * FROM todos
WHERE similarity(title, 'lern postgre') > 0.3;

-- Fuzzy matching operator
SELECT * FROM todos WHERE title % 'postgre';  -- % means "is similar to"

-- Create a GIN index for fast trigram search
CREATE INDEX idx_todos_title_trgm ON todos USING gin (title gin_trgm_ops);
```

> **Use case:** A search box where users might misspell words. `pg_trgm` finds matches even with typos.

## 3. pgvector — Vector Similarity Search

`pgvector` adds a `vector` type and similarity search operators. This is how you do **semantic search** (search by meaning, not just keywords).

### What is a vector?

A vector is an array of numbers (called an **embedding**) that represents the meaning of text. Similar texts have similar vectors.

```
"Buy groceries"    → [0.12, -0.45, 0.78, ...]   (384 numbers)
"Get food from store" → [0.11, -0.43, 0.76, ...]   (very similar!)
"Learn PostgreSQL"  → [-0.67, 0.23, 0.01, ...]   (very different)
```

Vectors are generated by an AI model (like OpenAI's `text-embedding-3-small`). You store them in PostgreSQL, then find similar rows.

### Setup

```sql
CREATE EXTENSION IF NOT EXISTS vector;

-- Add a vector column to a table
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    content TEXT NOT NULL,
    embedding vector(384)    -- 384 dimensions (depends on the model)
);
```

### Insert vectors

```sql
-- Vectors are inserted as arrays
INSERT INTO documents (content, embedding)
VALUES ('Learn PostgreSQL basics', '[0.12, -0.45, 0.78, ...]');
-- In practice, you generate embeddings via an API, not by hand
```

### Similarity search

pgvector supports three distance operators:

| Operator | Name | Use when |
|----------|------|----------|
| `<->` | L2 distance (Euclidean) | Default choice |
| `<=>` | Cosine distance | Normalized embeddings (most common with AI models) |
| `<#>` | Inner product (negative) | Already-normalized, need speed |

```sql
-- Find the 5 most similar documents to a query vector
SELECT content, embedding <=> '[0.11, -0.43, 0.76, ...]' AS distance
FROM documents
ORDER BY embedding <=> '[0.11, -0.43, 0.76, ...]'
LIMIT 5;
```

### Indexing for performance

Without an index, pgvector does exact (brute-force) search. For large datasets, add an index:

```sql
-- IVFFlat index (faster, approximate)
CREATE INDEX idx_documents_embedding ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);
-- lists = sqrt(number of rows) is a good starting point

-- HNSW index (faster queries, slower to build, more accurate)
CREATE INDEX idx_documents_embedding ON documents
USING hnsw (embedding vector_cosine_ops);
```

| Index type | Build speed | Query speed | Accuracy | Memory |
|-----------|-------------|-------------|----------|--------|
| None (exact) | N/A | Slow | 100% | Low |
| IVFFlat | Fast | Medium | ~95% | Low |
| HNSW | Slow | Fast | ~99% | High |

> **For your todos app** with small data, no index is needed. Add one when you have thousands of rows.

### Full example with your todos app

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE todos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title TEXT NOT NULL,
    description TEXT,
    embedding vector(384)
);

-- After generating embeddings via an API:
INSERT INTO todos (title, description, embedding)
VALUES ('Buy groceries', 'Get milk, eggs, and bread', '[0.12, ...]');

-- Semantic search: "food shopping"
SELECT title, description,
    embedding <=> '[0.11, ...]' AS distance
FROM todos
WHERE embedding IS NOT NULL
ORDER BY embedding <=> '[0.11, ...]'
LIMIT 5;
```

### Supabase + pgvector

Supabase has pgvector pre-installed. You can:
1. Store embeddings in your table
2. Create an Edge Function that calls an embedding API (OpenAI, etc.)
3. Query with the Supabase client using `.rpc()` to call a similarity search function

```sql
-- A Supabase RPC function for semantic search
CREATE OR REPLACE FUNCTION search_todos(
    query_embedding vector(384),
    match_threshold FLOAT DEFAULT 0.8,
    match_count INT DEFAULT 5
)
RETURNS TABLE (id UUID, title TEXT, description TEXT, similarity FLOAT) AS $$
    SELECT id, title, description,
        1 - (embedding <=> query_embedding) AS similarity
    FROM todos
    WHERE embedding IS NOT NULL
        AND 1 - (embedding <=> query_embedding) > match_threshold
    ORDER BY embedding <=> query_embedding
    LIMIT match_count;
$$ LANGUAGE sql;

-- Call from Supabase client:
-- supabase.rpc('search_todos', { query_embedding: [...], match_count: 5 })
```

## 5. Extensions in Migrations

Always enable extensions in your first migration:

```sql
-- supabase/migrations/00001_enable_extensions.sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
```

`IF NOT EXISTS` makes it safe to run multiple times.

## Practice

```sql
-- 1. Enable extensions
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS vector;

-- 2. Create a table with both text search and vector search
CREATE TABLE articles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title TEXT NOT NULL,
    body TEXT NOT NULL,
    embedding vector(3)   -- using 3 dimensions for easy testing
);

-- 3. Insert test data with tiny vectors
INSERT INTO articles (title, body, embedding) VALUES
    ('PostgreSQL basics', 'Learn SQL with PostgreSQL', '[1, 0, 0]'),
    ('Advanced PostgreSQL', 'Triggers, functions, and more', '[0.9, 0.1, 0]'),
    ('Learn Python', 'Python programming tutorial', '[0, 0, 1]'),
    ('Database design', 'Schema design best practices', '[0.8, 0.2, 0.1]');

-- 4. Trigram search: find articles similar to "postgre"
SELECT title, similarity(title, 'postgre') AS sim
FROM articles
WHERE title % 'postgre'
ORDER BY sim DESC;

-- 5. Vector search: find articles most similar to [1, 0, 0] (PostgreSQL basics)
SELECT title, embedding <=> '[1, 0, 0]' AS distance
FROM articles
ORDER BY embedding <=> '[1, 0, 0]'
LIMIT 3;

-- 6. Combine both: semantic search with fuzzy title filter
SELECT title, embedding <=> '[0.85, 0.15, 0]' AS distance
FROM articles
WHERE title % 'postgres' OR embedding <=> '[0.85, 0.15, 0]' < 0.5
ORDER BY distance
LIMIT 3;

-- Cleanup
DROP TABLE articles;
DROP EXTENSION pg_trgm;
DROP EXTENSION vector;
```
